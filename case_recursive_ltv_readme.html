RecursiveLTV

Production-first LTV forecasting engine for mobile gaming UA/marketing data science.

Features

  • Direct MMP API ingestion with rolling backfill sync.
  • Recursive step-chain prediction: D0 -> D1 -> D3 -> D7 -> D14 -> D30.
  • Power-law extrapolation D31 -> D365.
  • Hierarchical backoff for scaling factors: campaign -> network -> platform -> game -> global.
  • Confidence and anomaly scoring with WAPE gates.
  • Flask + Chart.js dashboard with emergency refresh control.
  • Channel bucket filter (Non-Incent Channels vs Incent Channels) and target-ROAS KPI panel.
  • CPI-primary ROAS accuracy layer with audit ROAS preservation, hierarchical CPI imputation, and quality flags.
  • Model quality and forecast reliability layer with champion-challenger selection and p10/p50/p90 bands.
  • UA Decision Cockpit with Scale/Hold/Cut actions, conservative guardrails, impact-first ranking, prior-window deltas, CSV export, and shareable URL state.
  • Backtesting & Decision Support snapshots with deterministic bootstrap simulator and advisory threshold calibration.

Quick start

  1. Create venv and install dependencies:
python -m venv .venv
source .venv/bin/activate
pip install -e .[dev]
  1. Configure env:
cp .env.example .env
  1. Run DB migrations:
alembic upgrade head
  1. Start app:
flask --app app run --debug

Daily sync

python scripts/run_daily_sync.py --run-mode incremental_daily

If the dashboard is empty: - Run Emergency Refresh from UI with a wider Sync Window (days) (up to 1080), or - run sync manually with python scripts/run_daily_sync.py --window-days 1080.

Sync behavior note: - Sync Window (days) is ingestion-only and independent from dashboard Install Start/Install End filters. - dashboard date filters control read/query views only; they do not limit MMP ingestion window.

Use sync admin endpoints for operational visibility: - GET /api/admin/sync/status for latest run + active run fields (is_running, active_run_id, active_stage_name). - GET /api/admin/sync/runs for paginated run history. - GET /api/admin/sync/runs/<run_id>/stages for stage timeline. - GET /api/admin/storage/status for table counts, retention cutoffs, and reclaimable-row estimates. - POST /api/admin/storage/maintenance to run retention/compaction without ingestion (requires sync.run).

Pipeline robustness

  • Sync pipeline now records stage-level telemetry in sync_run_stages (migration 0010_pipeline_robustness_sync_stages).
  • Stage data is exposed from GET /api/admin/sync/status as last_run_stages with:
  • stage_name, status, started_at, completed_at, duration_ms
  • details (JSON payload for stage-specific counters)
  • error_message (if failed)
  • Daily sync response now includes normalization counters:
  • rows_normalized
  • rows_skipped_invalid
  • normalize_skip_reasons
  • stage_count
  • MMP ingestion now retries transient failures (429/5xx and network exceptions) using:
  • MMP_RETRY_ATTEMPTS
  • MMP_RETRY_BACKOFF_SECONDS
  • Large MMP date windows are automatically chunked (default 90 days) via:
  • MMP_FETCH_CHUNK_DAYS
  • Chunk requests can run in parallel for faster long-window sync via:
  • MMP_FETCH_PARALLEL_CHUNKS (default 4)

Storage retention policy

  • Detail retention is bounded by DETAIL_RETENTION_DAYS (default 365).
  • Cohorts older than detail TTL but inside MAX_BACKFILL_DAYS are compacted to milestone horizons from METRIC_COMPACTION_MILESTONES.
  • Cohorts older than MAX_BACKFILL_DAYS are pruned with cascade cleanup.
  • Operational and decision-support history is pruned via:
  • RUN_HISTORY_RETENTION_DAYS
  • BACKTEST_DETAIL_RETENTION_DAYS
  • MODEL_QUALITY_RETENTION_DAYS
  • AUTH_AUDIT_RETENTION_DAYS
  • Incremental/frequent ingestion and weekly reconcile windows:
  • INCREMENTAL_FETCH_DAYS
  • FULL_RECONCILE_DAYS
  • Weekly maintenance optimization can be toggled with:
  • RETENTION_VACUUM_WEEKLY
  • Maintenance stages are written to sync stage telemetry, including:
  • retention_dedupe, retention_compaction, retention_prune_runs,
  • retention_prune_backtests, retention_prune_quality_tables,
  • retention_prune_auth_logs, and optional retention_vacuum_optimize.

MMP adapter

  • Schema adapter docs: MMP_SCHEMA_ADAPTER.md
  • Supports flat, wrapped, nested dimensions/metrics, and columns+rows payload shapes
  • Controlled by MMP_* mapping variables in .env
  • Uses MMP RS JSON report endpoint (/reports-service/report) with Authorization: Bearer <token>, date_period, and app_token__in.
  • To align with Datascape views, configure explicit report filters in .env:
  • MMP_UTC_OFFSET (example +00:00)
  • MMP_ATTRIBUTION_SOURCE (example first)
  • MMP_AD_SPEND_MODE (example network)
  • MMP_COHORT_MATURITY (example immature)
  • MMP_AD_REVENUE_SOURCES (example Applovin MAX FB only,AppLovin Max)
    • When this is set, the API pull includes the same ad_revenue_sources filter.
  • MMP_APPLY_AD_REVENUE_SOURCE_FILTER is a legacy toggle and no longer required.
  • After any MMP_* env change, restart Flask/Gunicorn and run a new sync.
  • For ROAS parity, include roas_d* metrics in MMP_METRICS and keep MMP_ROAS_METRIC_PREFIXES=actual_roas_d,roas_d.
  • Include install metric in pulls for CPI/ROAS derivation:
  • strict bucket policy:
    • non_incent cohorts use network_installs
    • incent cohorts use installs
  • alias accepted by adapter config: installs_network (auto-mapped to network_installs)
  • configure field aliases with:
    • MMP_FIELD_INSTALLS_NETWORK
    • MMP_FIELD_INSTALLS
  • Include cost metrics in pulls for bucket-specific spend parity:
  • strict bucket policy:
    • non_incent cohorts use network_cost
    • incent cohorts use mmp_cost
  • configure field aliases with:
    • MMP_FIELD_COST_NETWORK
    • MMP_FIELD_COST_MMP
  • Include extended actual checkpoints in MMP_METRICS:
  • lifetime_value_d60,lifetime_value_d90,lifetime_value_d120
  • If MMP_ENDPOINT_PATH=/cohorts is still configured, the client auto-retries with /report.

Auth

  • Session-based login is enabled via /login with DB-backed users.
  • Set FLASK_SECRET_KEY in .env (required for secure sessions).
  • Session hardening controls:
  • SESSION_COOKIE_SECURE
  • SESSION_IDLE_TIMEOUT_MINUTES
  • SESSION_ABSOLUTE_TIMEOUT_HOURS
  • Login protection controls:
  • AUTH_MAX_FAILED_ATTEMPTS
  • AUTH_LOCKOUT_MINUTES
  • Password policy controls:
  • PASSWORD_MIN_LENGTH
  • PASSWORD_REQUIRE_LETTER
  • PASSWORD_REQUIRE_DIGIT
  • PASSWORD_REQUIRE_SPECIAL
  • Bootstrap flow: when no users exist, open /login and create the first admin user.
  • User/permission management is available in dashboard panel: create, delete, set role, set explicit permissions.
  • Password rotation API:
  • POST /api/auth/change-password (current_password, new_password)
  • Auth audit API (admin):
  • GET /api/admin/auth/audit?limit=100&action=login&username=admin

Roles: - admin: all permissions (*) - operator: dashboard.view, sync.run, secrets.view - viewer: dashboard.view

Known permissions: - dashboard.view - sync.run - secrets.view - secrets.manage - users.manage - taxonomy.manage
