Stack & Tooling

Last updated: 2026-02-27

The engine is pure Python. That wasn't a deliberate minimalism call — it was a deployment constraint. The MMP API client and the forecasting logic needed to run on the same machine without a separate build step, and the data engineering environment at the studio was Python-first. Adding another runtime would have meant a separate deployment pipeline no one wanted to own.

Core dependencies

numpy and pandas throughout, which is unremarkable. The more considered choice was scipy.optimize for curve fitting rather than scikit-learn. Power-law LTV curves require bounded minimization over a specific functional form — scipy.optimize.curve_fit exposes the bounds and the residual function directly. sklearn's pipeline abstractions would have added indirection without adding value; when you're fitting LTV(t) = a · t^b with business-constraint bounds on a and b, you want to be close to the math.

requests handles MMP API calls. The client does exponential-backoff retry on 429 and 5xx, and chunks large date windows into 90-day slices to stay within rate limits.

openpyxl is an optional dependency for Excel backtest report output. If it's not installed the engine falls back to CSV.

What's absent and why

No test suite in the repo. That's intentional. The backtesting framework — fit the model on data up to day D, predict forward, compare against actuals — serves as the validation layer. It's slower than unit tests but it validates the thing that matters: whether the predictions hold up against real historical data. Utility functions (date arithmetic, reshape_long) have unit tests; the model itself is validated through backtests.

No pyproject.toml or requirements.txt. This started as a script that grew into production use. Dependencies are numpy, pandas, scipy, requests, and optionally openpyxl.

No web framework. The engine is a command-line batch tool driven by argparse. The CLI was added when the model needed to run on a schedule; before that it was called directly as a function.
